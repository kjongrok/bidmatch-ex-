## 5. 상세 기능 요구사항 (Functional Requirements)

### 5.1 회원 관리 (Auth)
* **FE-01 (회원가입):** 이메일, 비밀번호, 사용자명을 입력받아 가입을 진행한다. (이메일 중복 체크 필수)
* **FE-02 (로그인):** 이메일과 비밀번호를 검증하여 성공 시 JWT 토큰을 발급하며, 로컬 스토리지에 보관한다.
* **BE-01 (인증 권한 검증):** 관심 키워드 등록/삭제, 알림 조회의 경우 HTTP Header에 `Authorization: Bearer {JWT}`가 유효한지 체크한다.

### 5.2 발주 정보 조회 및 검색 (Project)
* **FE-03 (목록 및 검색):** 데이터베이스에 적재된 발주 정보를 최신순으로 리스팅하며, **사업명, 기관명, 예산 범위** 필터를 통해 클라이언트가 검색 요청을 보낼 수 있다.
* **FE-04 (상세 조회):** 특정 발주 사업을 클릭 시 사업번호, 예산, 공고일자 등 공공데이터 원본 상세 내용을 모달 또는 팝업으로 제공한다.
* **BE-02 (페이징 처리):** 대량의 발주 데이터 조회를 고려하여 백엔드는 인덱싱이 적용된 Page, Size 기반 페이징 API를 구현한다.

### 5.3 관심 키워드 관리 (Keyword)
* **FE-05 (키워드 CRUD):** 사용자는 본인의 대시보드에서 매칭을 원하는 키워드(예: `클라우드`, `블록체인`, `웹진`)를 등록 및 삭제할 수 있다. (인당 최대 10개 제한)
* **BE-03 (유효성 검사):** 공백이나 특수문자로만 이루어진 키워드는 등록을 거부한다.

### 5.4 배치 및 매칭 알림 (Batch & Notification)
* **BE-04 (공공데이터 수집 배치):** APScheduler가 주기적으로 나라장터 발주계획 API를 호출하여 데이터를 파싱한다. `project_no(사업번호)`를 유니크 키로 잡아 이미 존재하는 데이터는 건너뛰고 신규 데이터만 DB에 적재한다(**Upsert 로직**).
* **BE-05 (키워드 매칭 및 SMTP 발송):** 신규 적재된 사업명의 텍스트 정보와 사용자들이 등록한 관심 키워드를 `LIKE` 유연 검색으로 매칭한다. 매칭된 사용자의 이메일 주소로 SMTP 라이브러리를 이용해 발주 알림 메일을 즉시 발송한다.

## 10. MariaDB 연동 및 배포 핵심 체크포인트 (DevOps 가이드)

### 1) DB 드라이버 및 SQL Alchemy 접속 URI 설정

* **체크포인트:** Python 환경에서 MariaDB와 통신하기 위해 `pymysql` 라이브러리가 필수적입니다.
* **해결 방안:** `requirements.txt`에 `pymysql`을 반드시 추가하고, Flask의 `SQLALCHEMY_DATABASE_URI`를 아래와 같은 형식으로 설정합니다.
```python
mysql+pymysql://username:password@mariadb-host-address:3306/dbname

```



### 2) 외부 Cloud DB 자원 확보 (데이터 휘발 방지)

* **체크포인트:** Render 무료 플랜은 내부 디스크 저장소(Volume)가 유료입니다. 만약 Render 백엔드 내부에 Docker로 MariaDB를 가동하면 **서버가 재부팅될 때마다 DB 데이터가 전부 삭제**됩니다.
* **해결 방안:** DB의 지속성 확보를 위해 외부 무료 클라우드 MariaDB/MySQL 호스팅(Aiven 또는 Railway 등)에서 인스턴스를 단독 생성한 뒤, 주소값만 Render 백엔드에 연동합니다.

### 3) Render Cold Start로 인한 배치(APScheduler) 정지 문제 해결

* **체크포인트:** Render 무료 플랜 백엔드는 **15분간 외부 요청이 없으면 서버가 정지(Sleep)** 상태로 전환됩니다. 서버가 잠들면 Flask 내부에서 작동하는 **공공데이터 수집 스케줄러(APScheduler)도 함께 멈추게 됩니다.**
* **해결 방안:** 외부 무료 크론 서비스(예: Cron-Job.org)에 가입한 후, 10분~13분 주기로 Flask 백엔드의 헬스체크 엔드포인트를 찌르도록 스케줄을 걸어 서버 활성화(Wake-up) 상태를 유지시킵니다.

### 4) 프론트엔드-백엔드 간 CORS 및 환경 변수 연동

* **체크포인트:** 프론트엔드(Static Site)와 백엔드(Web Service)의 도메인이 다르게 생성되므로 CORS 에러가 발생합니다.
* **해결 방안:**
* **Flask (백엔드):** `flask-cors` 라이브러리를 통해 프론트엔드의 Render 배포 주소를 명시적으로 허용합니다.
* **React (프론트엔드):** Vite 환경변수(`.env.production`) 파일에 백엔드 배포 주소를 `VITE_API_BASE_URL`로 기입하고, Axios 인스턴스 생성 시 이를 기준 주소(BaseURL)로 사용합니다.
