## 1. 서비스 목표

- 나라장터 공공 API 데이터를 자동 수집한다.
- 사용자가 등록한 조건과 일치하는 공고를 자동으로 선별한다.
- 매칭된 공고를 이메일로 알림 전송한다.
- 사용자가 직접 나라장터에서 반복 검색하는 시간을 줄인다.
- 공고 마감일, 관심 키워드, 기업 조건을 기준으로 놓치기 쉬운 공고를 쉽게 확인하게 한다.
- 공고 데이터의 물품분류, 기관, 일정, 금액 정보를 구조화해 검색과 추천에 활용한다.

## 2. 핵심 사용자

- 공공입찰 정보를 정기적으로 확인해야 하는 사업자 또는 담당자
- SI 기업, 스타트업, 프리랜서 개발자
- 특정 키워드, 기관, 지역, 업종, 예산 조건에 맞는 공고만 확인하고 싶은 사용자
- 청년기업, 여성기업, 장애인기업 등 정책 조건에 해당하는 기업 사용자
- 연령, 성별, 지역, 관심분야 등 개인 특성에 맞는 공고 추천을 받고 싶은 사용자
- 수집/알림 상태를 관리해야 하는 관리자

## 3. MVP 필수 요구사항

### 3.1 회원 및 인증

- 사용자는 이메일, 비밀번호, 사용자명으로 회원가입할 수 있어야 한다.
- 이메일 중복 가입을 방지해야 한다.
- 사용자는 이메일과 비밀번호로 로그인할 수 있어야 한다.
- 로그인 성공 시 인증 토큰을 발급해야 한다.
- 인증이 필요한 API는 요청 헤더의 토큰을 검증해야 한다.
- 사용자는 로그아웃할 수 있어야 한다.
- 사용자는 비밀번호를 재설정할 수 있어야 한다.
- 사용자는 기본 프로필 정보를 조회하고 수정할 수 있어야 한다.
- 사용자의 약관, 개인정보, 이메일 수신 동의 이력을 저장해야 한다.

### 3.2 사용자/기업 정보

- 사용자는 회사명 또는 표시명을 등록할 수 있어야 한다.
- 사용자는 연령, 성별, 지역, 관심분야 등 기본 정보를 등록할 수 있어야 한다.
- 사용자는 업종 정보를 등록할 수 있어야 한다.
- 사용자는 보유 면허 정보를 등록할 수 있어야 한다.
- 사용자는 관심 키워드를 등록할 수 있어야 한다.
- 사용자는 청년기업, 여성기업, 장애인기업 여부를 설정할 수 있어야 한다.
- 기업 조건 정보는 공고 추천 또는 필터링에 활용되어야 한다.
- 사용자 검색 이력은 추천 검색어 개선에 활용될 수 있어야 한다.

### 3.3 관심 조건 및 구독 관리

- 사용자는 관심 조건을 등록, 조회, 수정, 삭제할 수 있어야 한다.
- 관심 조건에는 포함 키워드를 설정할 수 있어야 한다.
- 관심 조건에는 제외 키워드를 설정할 수 있어야 한다.
- 관심 조건에는 업무 구분 또는 사업 유형을 설정할 수 있어야 한다.
- 관심 조건에는 기관명을 설정할 수 있어야 한다.
- 관심 조건에는 지역 조건을 설정할 수 있어야 한다.
- 관심 조건에는 업종 조건을 설정할 수 있어야 한다.
- 사용자는 관심 조건을 활성화하거나 비활성화할 수 있어야 한다.
- 사용자는 알림 수신을 해지하거나 재개할 수 있어야 한다.
- 초기 알림 빈도는 일 1회로 제공한다.
- 관심 키워드는 사용자별 최대 개수를 제한할 수 있다.

### 3.4 나라장터 데이터 수집

- 백엔드는 나라장터 공공 API를 호출해 공고 데이터를 수집해야 한다.
- 백엔드는 나라장터 표준연계 서비스의 입찰물품공고서를 API로 검색하고 저장해야 한다.
- 수집 작업은 주기적으로 실행되어야 한다.
- MVP 기준 최소 일 1회 수집을 지원한다.
- 최근 1~3일 범위의 보정 수집을 지원해야 한다.
- 원본 API 응답은 Raw 데이터로 저장해야 한다.
- 서비스에서 사용하는 공고 정보는 정규화된 테이블에 저장해야 한다.
- 공고 고유 키를 기준으로 중복 저장을 방지해야 한다.
- 이미 존재하는 공고는 갱신하고, 신규 공고는 추가해야 한다.
- 업무 구분, 업무 여부, 입찰공고번호, 공고명, 공고기관, 수요기관, 게시일시를 관리해야 한다.
- 공고명, 공고일자, 개찰일자, 공고 종류, 입찰 금액을 검색과 필터 기준으로 관리해야 한다.
- 조달데이터허브에서 제공하는 물품분류체계를 활용해 공고를 분류할 수 있어야 한다.
- 수집 실행 로그를 저장해야 한다.
- 수집 로그에는 시작 시각, 종료 시각, 상태, 수집 건수, 저장 건수, 오류 메시지가 포함되어야 한다.
- API 응답 형식이 변경되어도 원인 분석이 가능하도록 원본 응답을 보관해야 한다.
- 업무시간 기준 09:00~18:00 사이에는 1시간당 1회 정기 데이터 갱신을 지원할 수 있어야 한다.
- 매일 08:00 일괄 배치를 통해 데이터를 갱신할 수 있어야 한다.
- 관리자가 재조회 기능을 실행해 추가된 내용을 즉시 반영할 수 있어야 한다.

### 3.5 공고 조회 및 검색

- 사용자는 수집된 공고 목록을 조회할 수 있어야 한다.
- 공고 목록은 최신순으로 조회할 수 있어야 한다.
- 공고 목록은 페이지네이션을 지원해야 한다.
- 사용자는 공고 제목 또는 사업명으로 검색할 수 있어야 한다.
- 사용자는 공고명, 공고일, 입찰일을 기준으로 검색할 수 있어야 한다.
- 사용자는 기관명으로 검색할 수 있어야 한다.
- 사용자는 예산 범위로 검색할 수 있어야 한다.
- 사용자는 입찰 금액 기준으로 검색할 수 있어야 한다.
- 사용자는 지역, 업종, 업무 구분 조건으로 공고를 필터링할 수 있어야 한다.
- 사용자는 물품분류 기준으로 공고를 필터링할 수 있어야 한다.
- 사용자는 공고 상세 정보를 조회할 수 있어야 한다.
- 공고 상세에는 공고번호, 제목, 기관명, 예산, 공고일, 마감일, 원본 링크가 포함되어야 한다.
- 검색 결과는 CSV 파일로 다운로드할 수 있어야 한다.

### 3.6 매칭 로직

- 백엔드는 수집된 공고와 사용자 관심 조건을 비교해야 한다.
- 포함 키워드가 공고 제목 또는 주요 텍스트에 포함되면 매칭 후보로 판단한다.
- 제외 키워드가 포함된 공고는 매칭 결과에서 제외해야 한다.
- 기관, 지역, 업종, 업무 구분 조건을 매칭에 반영해야 한다.
- 기업 유형 조건이 설정된 경우 청년기업, 여성기업, 장애인기업 관련 공고 필터링에 활용해야 한다.
- 사용자 관심분야, 연령, 지역, 성별과 연계된 검색 조건을 추천에 활용할 수 있어야 한다.
- 회원 특성에 맞는 추천 검색어를 제공할 수 있어야 한다.
- 사용자의 검색 이력을 추천 검색어에 반영할 수 있어야 한다.
- 검색 조건과 공고 간 일치도를 계산해 점수로 보여줄 수 있어야 한다.
- 동일 사용자에게 동일 공고를 중복 알림하지 않아야 한다.
- 매칭 결과와 매칭 사유를 저장해야 한다.

### 3.7 이메일 알림

- 매칭된 공고가 있는 사용자에게 이메일을 발송해야 한다.
- MVP 기준 이메일 알림을 우선 제공한다.
- 알림은 일간 Digest 형태를 기본으로 한다.
- 알림 메일에는 매칭된 공고 목록이 포함되어야 한다.
- 알림 메일에는 공고 제목, 기관명, 마감일, 원본 링크가 포함되어야 한다.
- 이메일 발송 성공 여부를 저장해야 한다.
- 이메일 발송 실패 사유를 저장해야 한다.
- 이메일 발송 실패 건은 최소 1회 재시도할 수 있어야 한다.
- 사용자는 본인의 알림 발송 이력을 조회할 수 있어야 한다.

### 3.8 마감일 및 달력

- 사용자는 관심 공고의 마감일을 확인할 수 있어야 한다.
- 오늘 날짜 기준 마감이 임박한 공고를 확인할 수 있어야 한다.
- 관심 등록된 공고 또는 매칭 공고를 달력 형태로 볼 수 있어야 한다.
- 달력에는 공고 제목과 마감일이 표시되어야 한다.

### 3.9 관리자 기능

- 관리자는 사용자 목록을 조회할 수 있어야 한다.
- 관리자는 사용자 구독 상태를 조회할 수 있어야 한다.
- 관리자는 회원의 기본 정보와 검색 조건을 관리할 수 있어야 한다.
- 관리자는 물품분류 또는 물품 관련 기준 정보를 관리할 수 있어야 한다.
- 관리자는 데이터 수집 실행 현황을 조회할 수 있어야 한다.
- 관리자는 이메일 발송 현황을 조회할 수 있어야 한다.
- 관리자는 데이터 품질 지표를 조회할 수 있어야 한다.
- 데이터 품질 지표에는 중복 의심, 누락 의심, 파싱 실패가 포함되어야 한다.
- 관리자는 간단한 운영 공지를 관리할 수 있어야 한다.

## 4. 선택 요구사항

### 4.1 알림 채널 확장

- 카카오 알림톡 발송을 추가할 수 있다.
- 알림 주기를 실시간, 일간, 주간으로 세분화할 수 있다.

### 4.2 추천 고도화

- 사용자 조건에 맞는 키워드 추천 또는 템플릿을 제공할 수 있다.
- 공고별 관련도 점수를 계산할 수 있다.
- 계절, 날씨 등 외부 특징과 공고 데이터의 연관성을 분석할 수 있다.
- 공고문 핵심 내용을 3줄로 요약할 수 있다.
- 공고를 홍보, 마케팅, IT, 디자인 등 카테고리로 자동 분류할 수 있다.
- 유사 공고를 묶어 중복 노출을 줄일 수 있다.
- 기관, 업무 구분, 키워드 증감 추이를 리포트로 제공할 수 있다.

### 4.3 입찰 참여 제한 정보

- 사용자는 품목 키워드로 공고를 검색할 수 있다.
- 대기업 참여가 제한된 공고는 목록에서 구분 표시할 수 있다.
- 제한 사유 또는 관련 조건을 상세 화면에서 확인할 수 있다.

### 4.4 부정당업자 정보

- 부정당업자로 지정된 기업 정보를 조회할 수 있다.
- 공고 또는 기업 정보에 부정당업자 관련 위험 표시를 할 수 있다.
- 기업 상세에서 제재 사유를 확인할 수 있다.

### 4.5 소셜 로그인

- 사용자는 소셜 계정으로 로그인할 수 있다.

## 5. 데이터 요구사항

- 사용자 정보를 저장해야 한다.
- 사용자 프로필과 기업 정보를 저장해야 한다.
- 사용자 연령, 성별, 지역, 관심분야 정보를 저장할 수 있어야 한다.
- 사용자 검색 이력을 저장할 수 있어야 한다.
- 사용자 관심 조건을 저장해야 한다.
- 사용자 동의 이력을 저장해야 한다.
- 나라장터 원본 API 응답을 저장해야 한다.
- 정규화된 공고 정보를 저장해야 한다.
- 공고 종류, 개찰일자, 입찰금액, 물품분류 정보를 저장해야 한다.
- 조달데이터허브 물품분류체계 정보를 저장하거나 참조할 수 있어야 한다.
- 데이터 수집 실행 로그를 저장해야 한다.
- 공고 매칭 결과를 저장해야 한다.
- 이메일 발송 이력을 저장해야 한다.
- 중복 방지를 위한 공고 고유 키를 관리해야 한다.

- 회원가입 화면이 필요하다.
- 비밀번호 재설정 화면이 필요하다.
- 사용자 대시보드가 필요하다.
- 관심 조건 관리 화면이 필요하다.
- 공고 목록 화면이 필요하다.
- 공고 상세 화면이 필요하다.
- 마감일 달력 화면이 필요하다.
- 공고 검색 결과 CSV 다운로드 기능이 필요하다.
- 추천 검색어 표시 영역이 필요하다.
- 알림 이력 화면이 필요하다.
- 사용자 프로필/기업 정보 설정 화면이 필요하다.
- 관리자 대시보드가 필요하다.
- 관리자 수집 로그 화면이 필요하다.
- 관리자 알림 발송 로그 화면이 필요하다.

## 8. 비기능 요구사항

- 배포 URL은 상시 접속 가능해야 한다.
- API 키, DB 비밀번호, SMTP 비밀번호 등 시크릿은 환경 변수로 관리해야 한다.
- 수집 실패 시 원인을 로그로 확인할 수 있어야 한다.
- 이메일 발송 실패 시 실패 사유를 확인할 수 있어야 한다.
- 공고 중복 저장을 방지해야 한다.
- 대량 공고 조회를 고려해 페이지네이션과 인덱스를 적용해야 한다.
- Raw 데이터, 정규화 데이터, 매칭 결과, 알림 이력 간 추적이 가능해야 한다.
- 스케줄러 실행 주기와 수동 재조회 실행 결과를 추적할 수 있어야 한다.
- 프론트엔드와 백엔드 도메인이 다를 경우 CORS를 설정해야 한다.
- 외부 API 장애 또는 응답 변경에 대비해 예외 처리를 해야 한다.

## 11. 정리 기준

- 회원가입/로그인/인증 관련 중복 요구사항은 `회원 및 인증`으로 통합했다.
- 관심 키워드, 구독, 필터 조건 관련 요구사항은 `관심 조건 및 구독 관리`로 통합했다.
- 나라장터 API 수집, 배치, Upsert, Raw 저장 요구사항은 `나라장터 데이터 수집`으로 통합했다.
- 공고 조회, 검색, 상세 보기 요구사항은 `공고 조회 및 검색`으로 통합했다.
- 키워드 매칭과 사용자 조건 비교 요구사항은 `매칭 로직`으로 통합했다.
- 이메일 알림, SMTP, 발송 로그, 재시도 요구사항은 `이메일 알림`으로 통합했다.
- 입찰물품공고서, 공고명/일자/금액/물품분류 관리 요구사항은 `나라장터 데이터 수집`과 `공고 조회 및 검색`으로 통합했다.
- 연령/성별/지역/관심분야, 검색 이력 기반 추천 요구사항은 기본 정보와 추천 고도화 요구사항으로 분리했다.
- 업무시간 시간당 갱신, 08시 일괄 배치, 재조회 기능은 데이터 수집 요구사항과 관리자 기능으로 통합했다.
- CSV 다운로드와 일치도 점수는 공고 검색 및 매칭/추천 요구사항에 반영했다.
- AI 요약, 자동 태깅, 관련도 점수, 트렌드 분석은 MVP 이후 선택 기능으로 분리했다.
- 카카오 알림톡, 부정당업자, 대기업 참여 제한 정보는 MVP 이후 확장 기능으로 분리했다.
